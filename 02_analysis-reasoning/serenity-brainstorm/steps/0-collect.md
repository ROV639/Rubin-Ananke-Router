# 0-collect · 数据采集

## 这一步要回答什么

把目标账号、事件或 live claim 的原文和元数据采集全；这一步只采集，不分析。

## 怎么做

推理模式：

1. 明确目标：账号、关键词、日期窗口、是否只看原创、是否需要回复。
2. 用 Grok/X 搜索能力多次分页采集，硬指标：
   - 最近原创 ≥50 条（跳纯转发，thread 展开全文）
   - 互动最高 ≥20 条历史帖
   - 回复 tab ≥20 条
   - 达不到必须标"覆盖不足：实际N条/目标M条"并说明原因（搜索 degraded / 账号帖量少 / 权限等）
   - 不准用"样本足够"跳过量化目标
3. 采集工具优先 `x-grok-build xsearch` 或 Hermes `x_search`；对账号拆解需多次分页 Latest + 高互动过滤 + `filter:replies`。
4. 每条记录保留：原文、发布时间、点赞/转发/回复数、是否带链接/图片/视频、URL。
5. 如果 `xsearch` 无法给出完整原文，可用已登录浏览器或 X GraphQL 时间线作为采集入口；仍要保留原文与元数据。
6. 采集完报总条数和覆盖范围（vs 硬指标），不分析。

执行模式：

1. 只有当选题依赖 live X 讨论时才执行本步。
2. 采集目标从“账号全貌”缩小为“近 7 天讨论、代表性高互动帖、反对意见”。
3. 输出信号，不直接输出题材；进入后续 source/verify/filter。

## 案例举证

黄金范例中，Grok 明确说自己通过 “x_keyword_search 多次分页调用 Latest 模式 + min_faves 筛选高互动帖 + filter:replies 获取回复” 读取了 “41 条唯一帖文”。

## 推理模式输出 / 执行模式输出

推理模式输出：

- 原文清单
- 元数据表
- 总条数
- 覆盖范围：置顶、最近原创、高互动、回复

执行模式输出：

- live 信号清单
- 代表性 X 链接
- 反方或质疑信号
- 进入 `2-verify` 的候选 claims

## 质量门禁

- `degraded=true` 或 `STATUS: DEGRADED` 的 X 结果必须弃用。
- 不准用训练知识冒充真实推文。
- 不准只读首条就概括整个账号。
- 分析必须等采集后再做。

## 失败案例

- 只看最近帖，漏掉历史高互动帖；纠正：补高互动过滤。
- 只读 thread 第一条；纠正：展开全文或标注未展开。
- 采集阶段就下结论；纠正：输出原文表，进入后续步骤。
